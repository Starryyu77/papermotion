# Trae Solo Native Workflow

## Core Decision

PaperMotion should be built as a Trae Solo native workflow, not as a separate automation backend.

In the 3-hour MVP, Trae Solo is the production cockpit:

```text
Trae Solo workspace
  -> reads paper/formula files
  -> creates structured mechanism artifacts
  -> maintains a production manifest
  -> creates storyboard artifacts
  -> creates PixVerse asset jobs
  -> writes and repairs Manim code
  -> runs terminal render commands
  -> previews the website
  -> manages review comments and iteration
```

The public website is the final demo surface. The workflow itself should visibly live inside the repo and be executable through Trae Solo.

## Why This Fits Trae Solo

Trae Solo's current product positioning supports this shape:

- It can use a unified project workspace where files, context, and task history stay connected.
- It has Code Mode for implementation work and MTC / More Than Coding mode for documents, research, reports, and structured deliverables.
- It can orchestrate terminal, editor, browser, docs, and connectors in one visual workspace.
- It supports parallel agent work across different tasks.

Sources:
- https://www.trae.ai/solo
- https://www.trae.ai/blog/new_solo_beta_0331?v=1
- https://www.trae.ai/blog/trae_solo_mobile_0506

## Solo-First Product Shape

For the MVP, PaperMotion is three things:

1. A repo template that tells Trae Solo exactly how to transform a paper/formula into video assets.
2. A set of prompt files and intermediate artifacts that make the workflow auditable.
3. A website that showcases the final video and the workflow steps.

This avoids building a premature backend. Instead, Trae Solo acts as the execution environment.

## How To Use Trae Solo's Native Abilities

### 1. Workspace As Source Of Truth

Open `/Users/starryyu/Downloads/trae/papermotion` as the Solo Desktop workspace.

All work should produce files in the repo:

- Analysis output goes into `examples/<demo>/mechanism_spec.json`.
- Global coordination goes through `examples/<demo>/production_manifest.json`.
- Storyboard output goes into `examples/<demo>/storyboard.md`.
- Precise scene specs go into `examples/<demo>/scene_specs/*.json`.
- PixVerse jobs go into `production_manifest.pixverse_jobs`.
- Render logs and repair notes go into `examples/<demo>/render_notes.md`.
- Website code goes into `site/`.
- Cross-task decisions go into `memory/active/current.md`.

Avoid leaving decisions only in chat.

### 2. MTC Mode For Non-Code Work

Use MTC mode for:

- Reading a paper excerpt or formula.
- Extracting the mechanism.
- Creating a symbol ledger.
- Writing the storyboard.
- Producing narration copy.
- Writing PixVerse prompts.
- Updating production manifest sections for intent, prerequisites, curriculum, script, rhythm, and PixVerse requests.
- Reviewing educational correctness.

Expected outputs are structured files, not prose-only replies.

### 3. Code Mode For Execution

Use Code Mode for:

- Creating the website.
- Writing Manim code.
- Running Manim render commands.
- Exporting keyframes/stills for PixVerse image-to-video.
- Downloading or placing PixVerse generated clips in site assets when available.
- Fixing Python/Manim errors.
- Running the local dev server.
- Browser-previewing the site.
- Final repository cleanup.

Code Mode should use the files generated by MTC mode as inputs.

### 4. Parallel Agents For Three-Hour Delivery

Run three Solo task tracks in parallel after the Orchestrator creates the production manifest:

Track A: Explanation Film Pipeline
- `input.md`
- `production_manifest.json`
- `mechanism_spec.json`
- `storyboard.md`
- `scene_specs/*.json`
- `attention_demo.py`
- `attention-demo.mp4`

Track B: PixVerse And Audio Pipeline
- `visual_asset_requests`
- `pixverse_jobs`
- `asset_manifest`
- `audio_manifest`
- PixVerse clips or executable job specs

Track C: Website And Demo Surface
- Site skeleton
- Video player
- Chapter rail
- Formula ledger UI
- Workflow timeline
- Final README

Sync points:
- 0:45 freeze storyboard and symbol colors.
- 1:15 freeze first PixVerse job spec.
- 1:35 embed first rough render or placeholder.
- 2:40 final QA.

### 5. Visible Review Loop

Use Solo's preview/review style as part of the workflow:

- Preview `mechanism_spec.json` before storyboard generation.
- Preview `storyboard.md` before Manim generation.
- Preview `production_manifest.pixverse_jobs` before PixVerse generation.
- Render Manim low quality first.
- Preview website before final assembly.
- Comment directly on artifacts and regenerate only the broken stage.

This matches the research-backed pattern: staged generation, human review, partial repair.

  Input: video + artifacts

## What The Demo Should Show About Solo

The website should not hide Solo. It should show the workflow as a strength:

- "Built as a Trae Solo workspace."
- "Every stage is a file-backed artifact."
- "Manim handles exact formulas."
- "PixVerse handles cinematic support assets and metaphor clips."
- "Human review happens before each irreversible render step."

This makes the demo more credible than claiming full automation.

## Acceptance Criteria

PaperMotion is Trae Solo native if:

- A teammate can open the repo in Trae Solo and see the next task from `solo/task-board.md`.
- Every stage has a repo file input and output.
- The website can be rebuilt from repo files.
- Manim render commands are documented and runnable from Solo's terminal.
- PixVerse is represented as a formal cinematic asset stage, not a hidden dependency.
- Exact math remains in Manim-owned render layers.
- The final website demonstrates the workflow, not only the final video.
